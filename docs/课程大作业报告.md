# 加密资产 AI 研究台 v3 项目报告

课程：人工智能应用实践（IB00126）

项目名称：加密资产 AI 研究台 v3

执行模式：监控 + 人工复核，不自动交易

团队规模：1 人

## 1. 项目背景与需求分析

加密资产市场具有高波动、高噪声、高信息不对称的特点。一个代币的价格变化往往同时受到市场结构、链上资金、社交热度、榜单曝光、资金费率、持仓量和流动性等因素影响。普通用户在面对这些信息时容易出现两个问题：第一，只看到热度或涨幅，忽视结构确认和风险约束；第二，在多个来源给出相似提示时重复加权，误以为“多源共振”一定代表高置信交易机会。

本项目希望解决的不是“自动炒币”，而是构建一个可解释、可复核、可回放的 AI 研究台。系统的目标是将多源数据转成统一证据模型，并按严格流水线完成候选发现、生命周期判断、市场环境判断、双模路由和人工复核。最终输出只允许是观察、观察池、趋势多候选、衰竭空候选、区间派发观察或否决，不允许自动下单。

项目的实际应用场景是：用户输入“分析 ORDI”之类的自然语言指令，系统在看板中定位代币，展示当前候选状态、证据链、AI 复核说明、手动执行卡和失效条件。用户可以查看为什么它进池、为什么没有触发、为什么被否决、哪些数据源降级、哪些风险需要人工确认。所有人工复核会落盘，后续可在回放中查看，用于离线阈值校准。

从课程角度，本项目是一个大语言模型增强的数据分析 Agent。它综合运用了 Prompt Engineering、AI Agent 多步决策、MCP/工具集成思想、RAG 式上下文检索、安全护栏和 Web 交互界面。为了避免“AI 黑箱”，系统将所有中间状态显式化，包括 Evidence、FreshnessState、ResonanceBreakdown、ScoreBreakdown、RouterDecision 和 ManualReviewFeedback。

需求可以进一步拆成三类。第一类是研究需求：系统要能把多个数据源统一成候选池，不能让用户在不同网页、交易所和链上工具之间手工切换。第二类是解释需求：系统不能只显示一个“买/卖”标签，而要说明每个结论来自哪些证据、哪些证据已经过期、哪些来源正在降级，以及哪些条件会让结论失效。第三类是安全需求：系统不能诱导自动交易，不能因为用户追问就绕过风控，也不能把社交热度包装成确定性交易机会。三类需求共同决定了本项目的核心边界：它是 AI 研究台，不是自动交易机器人。

## 2. 技术方案设计

系统采用 monorepo 结构，前端是 React/Vite，后端是 Fastify，核心策略逻辑拆成多个 TypeScript package。整体依赖方向固定为单向：

```text
Raw Source Data
  -> Evidence
  -> Freshness
  -> Fusion
  -> Lifecycle
  -> Regime
  -> Router
  -> Manual Review Output
  -> Feedback Store
```

这条依赖方向是项目最重要的工程约束。Lifecycle 不能被 Regime 反向修改，RouterDecision 不能回写 Regime，ManualReviewFeedback 不能影响当前在线路由。这样做可以防止自我强化循环：系统不能因为已经给出多头结论，就反过来强化趋势判断；也不能因为人工复核偏多，就让当前候选立即改变路由。

数据源分为五个 source family：`market_structure`、`onchain_flow`、`rank_aggregator`、`social_sentiment`、`risk_annotation`。其中 market_structure 具备最高优先级和最高 veto 权；social_sentiment、rank_aggregator 和 onchain_flow 可以产生共振加分，但不能替代结构确认。risk_annotation 只能降权或标记风险，不能单独给方向。

v3 相比 v2 的关键升级有四点。第一，跨源融合从“简单去重”改成“同家族去重 + 跨家族共振”。同一 sourceFamily 内按 sameFamilyDedupeKey 去重，只保留 strongest/freshest 为 primary；不同 family 之间保留独立维度，计算 capped resonance bonus。第二，TTL 从硬过期改成 `fresh / degrading / stale` 三态，degrading 期间权重平滑下降，stale 只用于展示和回放。第三，评分变成显式 ScoreBreakdown，包括 discoveryScore、confirmationScore、riskPenalty、corroborationBonus、resonanceBonus、regimeMultiplier、lifecycleMultiplier、vetoState 和 finalScore。第四，人工复核从 UI 终点变成可落盘、可回放、可做离线阈值校准的闭环。

前端看板对应后端公共接口：`/api/source-coverage` 展示源覆盖和降级状态，`/api/candidates` 展示候选列表，`/api/candidates/:symbol` 展示详情，`/api/candidates/:symbol/replay` 展示回放，`/api/ai/analyze` 调用 AI 复核助手，`/api/manual-review` 保存人工复核。前端采用中文专业交易终端风格，包含左侧筛选、中间候选、右侧 sticky 详情区、执行卡、AI 助手、证据页、回放时间线和人工复核表单。

AI 与 RAG 的结合方式采用“受约束上下文注入”，而不是让模型直接访问原始市场数据后自由判断。后端先用确定性流水线得到 Candidate，再把候选状态、ScoreBreakdown、ResonanceBreakdown、ManualReviewChecklist、活跃证据摘要和失效条件注入 Prompt。模型只负责把这些结构化上下文转成可读的人工复核说明。这样既保留了大语言模型的表达和总结能力，又避免它在交易方向上自由发挥。未启用本地 Ollama 时，系统仍然用同一份上下文生成确定性 fallback 结果，保证课堂环境没有 API Key 也能演示。

## 3. 实现细节与关键代码说明

核心证据模型定义在 `packages/core/src/types.ts`。Evidence 统一包含 symbol、source、sourceFamily、category、direction、timestamp、ttlMs、freshnessState、freshnessWeight、strength、confidence、sameFamilyDedupeKey、crossFamilyThemeKey 和 isPrimary。这样不同来源不再直接影响路由，而是先被标准化为可比较的证据。

新鲜度逻辑在 `packages/core/src/freshness.ts`。系统根据 timestamp、ttlMs 和 degradingStartRatio 计算三态状态。fresh 权重为 1，degrading 权重线性下降到默认 0.2，stale 权重归零。stale evidence 不参与 lifecycle、regime 和 router scoring，只能出现在解释、审计和回放中。

融合逻辑在 `packages/core/src/fusion.ts`。dedupeSameFamily 只处理同一 sourceFamily、同一 symbol、同一 sameFamilyDedupeKey 的重复事件；computeResonance 计算跨家族同向共振，但上限受配置控制。这个设计解决了 v2 中“同一热度事件被多个来源重复包装后重复加分”的问题，同时保留了市场结构、链上流、榜单和舆情各自的信息价值。

路由逻辑在 `packages/core/src/router-lana.ts` 和 `packages/core/src/router-skanda.ts`。Lana 是趋势多头路由，硬规则是没有 market_structure.confirmation 最高只能 watchlist，social/rank/trading-signal 不能单独触发多头，late_speculative、range_chop、range_distribution 和 data_degraded 下不得高置信做多。Skanda 是操纵衰竭空头路由，必须处于 blowoff_exhaustion，必须有明确 exhaustion structure，没有结构衰竭禁止 short_candidate。新增的 range_distribution 分支只允许输出 range_distribution_watch，不直接触发做空。

AI 复核助手在 `packages/assistant/src/index.ts`。它优先支持本地 Ollama，默认模型为 `qwen2.5:7b`，通过 `RESEARCH_ENABLE_LLM=1` 显式启用。如果本地模型不可用或未启用，系统自动退回 deterministic-policy，仍然能够根据候选、证据、人工清单和失效规则生成结构化复核说明。Prompt 中强制写入安全边界：禁止自动下单、禁止收益承诺、禁止绕过 stale/data_degraded/market_structure veto。输出契约固定为 summary、tradePlan、checklist、evidenceUsed 和 invalidationRules。

安全护栏同样在 assistant 包中实现。detectAdvisorSafety 会识别自动交易请求，例如“直接市价买入”“帮我下单”“全仓梭哈”等，并标记 auto_execution_request；也会识别提示词注入和越狱内容，例如“忽略所有规则”“泄露 system prompt”等，并直接阻断。这样系统能在课程安全测试中证明自己对输入攻击和危险输出有明确防护。

前端实现中特别加强了可操作性。用户搜索“分析 ORDI”后，不会只看到一个孤立分数，而是会看到候选卡、执行摘要、手动步骤、禁开条件、证据家族、回放时间线和 AI 助手说明。执行卡把不同路由翻译成实际检查动作：趋势多候选强调顺势、小仓验证、回踩不破和结构失效退出；衰竭空候选强调 blowoff_exhaustion、失败反抽、短拿快跑和关键位收复退出；观察类候选明确提示“不执行”。这种设计是为了避免学生或用户误把研究结果当成自动交易信号。

持久化在 `packages/storage`。SQLite 保存低频状态与决策，包括 source_runs、source_status、evidence_events、candidate_snapshots、router_decisions、decision_reasons、degraded_flags、manual_review_feedback 和 readiness_checks。高频 market_structure 流不直接全量塞进 SQLite，而是写入 append-only JSONL，并生成窗口快照。这种分层避免 SQLite 写锁和体积膨胀，也方便后续做 replay。

## 4. TDD-for-AI 实践

课程要求至少 10 个 pytest 测试，并覆盖至少 2 个层级。本项目提供两套测试体系。第一套是 TypeScript Vitest，覆盖核心包、适配器、存储、回放、复核、API 和 Web。第二套是 Python pytest，文件位于 `tests/requirements_test.py`，作为可执行需求规约使用。

pytest 规约通过 Python 调用 `scripts/requirements-spec.ts`，直接验证 TypeScript 核心逻辑。测试覆盖三个层级以上：工具/核心函数测试、Agent 行为测试、安全测试和集成式契约测试。具体用例包括：freshness_fresh、freshness_degrading、freshness_stale_blocks_routing、fusion_same_family_dedupes_primary、fusion_cross_family_resonates、router_discovery_without_structure_is_watchlist、router_structure_unlocks_long、router_blowoff_exhaustion_short、safety_flags_auto_execution、safety_blocks_prompt_injection、advisor_fallback_is_manual_only 和 conflict_structure_beats_social。

这些测试不是形式化样例，而是直接表达系统需求。例如“discovery 很强但没有结构 confirmation 只能 watchlist”用测试断言 routerDecision；“stale structure 不得高置信路由”用测试断言 regimeState 降为 data_degraded 且不得输出 trend_long_candidate；“market_structure veto 高于 social heat”用测试断言社交热度不能覆盖结构否决。这样需求变化时，测试会第一时间暴露系统是否违背设计原则。

## 5. 测试结果与评估

项目的主要验证命令包括：

```bash
pnpm test
python -m pytest tests/requirements_test.py
pnpm build
```

当前测试重点不是预测收益，而是验证系统行为是否符合安全和工程约束。Freshness 测试证明证据会从 fresh 平滑进入 degrading，再进入 stale，且 stale 不参与当前路由。Fusion 测试证明同家族重复事件不会重复计分，跨家族事件不会被错误塌缩成单条证据。Router 测试证明 Lana 和 Skanda 都需要 market_structure confirmation，社交热度或榜单不能单独触发交易候选。Safety 测试证明系统会识别自动交易和提示词注入请求。Manual review 测试证明反馈可以落盘、回放，但不会回写当前 routing。

系统评估还包含可解释性检查。每个 Candidate 都必须输出 routerDecision、confidenceBand、decisionReason、riskFlags、degradedFlags、manualReviewRequired、scoreBreakdown 和 resonance。前端会把这些字段拆成分数仪表、证据家族、执行步骤、禁开条件和回放时间线。答辩 Demo 中，用户可以输入“分析 ORDI”，查看它为什么被归为趋势多候选、哪些证据是 confirmation、哪些数据源降级，以及 AI 助手如何给出人工复核框架。

安全方面，项目明确禁用自动交易、自动仓位管理、自动止盈止损和在线学习。即使用户要求“帮我直接下单”，AI 助手也只会标记风险并输出 manual-only 框架。即使用户要求“忽略所有规则”，系统也会阻断该请求。所有高置信输出都需要人工复核，且关键结构源 stale/unavailable/rate_limited 时，不允许输出高置信 long 或 short。

边界场景评估是本项目测试的重点。真实市场中最常见的问题不是没有信号，而是信号太多、互相污染或时间窗口不一致。因此测试专门覆盖了 social/rank 共振但无结构确认、结构源 stale、market_structure veto 与 social heat 冲突、自动交易诱导、提示词注入等场景。通过这些测试可以证明，系统在外部数据不稳定时会降级到 observe/watchlist，而不是强行给出高置信方向。这一点比单纯追求更高命中率更重要，因为课程项目需要展示 AI 系统工程化，而不是展示无法验证的收益曲线。

## 6. 总结与未来改进方向

本项目完成了一个可运行、可解释、可测试的 AI 加密资产研究台。它不是简单的数据看板，而是一个具备证据标准化、三态新鲜度、同家族去重、跨家族共振、生命周期分类、市场环境分类、双模路由、AI 复核、安全护栏和人工反馈闭环的系统。其核心价值在于把“我为什么看到这个标的”“它为什么能或不能交易”“哪些条件会让结论失效”显式呈现出来。

未来改进方向包括四点。第一，继续增强真实数据源，将 Binance skill、OKX OnchainOS、清算热力图、OI/funding 和订单簿数据接成更稳定的长连接。第二，引入更完整的 RAG 知识库，把策略文档、历史复盘、人工反馈和错误案例统一检索，提升 AI 助手的上下文质量。第三，增加更多 replay 数据，用历史趋势启动、meme 末端派发、blowoff 后衰竭和 range distribution 样本校准阈值。第四，扩展安全测试，例如越狱语料、误导性投资建议、自动交易诱导和数据降级场景。

项目仍然坚持不做自动交易。原因是课程项目的目标是 AI 应用系统与工程化实践，而不是实盘收益。将系统边界固定为 monitoring + manual review only，既符合金融风险控制，也更利于展示 AI Agent、Prompt Engineering、RAG、工具集成和 TDD-for-AI 的工程价值。
